    AWSIM Labs#

    AWSIM Labs 是 TIER IV/AWSIM 的一个分支，后者是使用 Unity 制作的开源模拟器，用于自动驾驶研发. 它是为自动驾驶软件开发的，如[Autoware](../autoware/ 的 Package. 该模拟器旨在弥合虚拟世界和现实世界之间的差距，使用户能够在安全可控的环境中训练和评估他们的自动驾驶系统，然后再将其部署到真实车辆上. 它为训练、测试和评估自动驾驶系统的各个方面提供了一个逼真的虚拟环境.

    AWSIM 使用精确的物理和传感器模型模拟各种真实场景. 它提供广泛的传感器，例如：摄像头、GNSS、IMU 和 LiDAR，使开发人员能够准确模拟其自动驾驶汽车与环境的交互. 该模拟器还对动态对象(如行人、其他车辆和交通信号灯)进行建模，从而可以研究复杂交通场景中的交互和决策. 这使得能够在不同的传感器配置和场景下测试和评估感知、规划和控制算法.

    AWSIM 支持灵活的模块化架构，便于自定义和扩展其功能. 用户可以使用自己的资产和交通规则修改当前环境或添加新环境，以创建自定义场景以满足其特定的研究需求. 这允许在各种驾驶条件下开发和测试高级算法.

    因为 AWSIM 主要是为了与 Autoware，它支持:

    • Ubuntu 22.04 和 Windows 10/11
    • ROS2 Humble 发行版

    WSL2 支持

    AWSIM Labs 不支持在适用于 Linux 2 的 Windows 子系统 (WSL2) 上运行. 建议使用原生 Ubuntu 22.04 环境.

    为什么要开发 AWSIM？#

    AWSIM 的主要目标是促进自动驾驶的研发，实现算法和系统的基准测试，并促进自动驾驶社区内的协作和知识交流. 通过提供一个逼真且可访问的平台，AWSIM 旨在加速自动驾驶领域的进步和创新.

    架构#

    要描述 AWSIM 的架构，首先需要提到 Environment. 它包含在特定场景的模拟中出现的所有对象及其配置. 为使用 Autoware 而开发的默认 AWSIM 场景称为 AutowareSimulation.

    在场景中，我们可以区分基本组件，例如 MainCamera、ClockPublisher、EventSystem 和 Canvas. 场景及其组件的详细描述可以在 查看这里 找到.

    除了上面提到的元素之外，该场景还包含另外两个非常重要且复杂的组件： Environment 和 EgoVehicle ——如下所述.

    Environment#

    Environment 是一个组件，其中包含所有模拟场景中环境的 Visual Elements 以及提供对环境的控制的元素. 它还包含两个组件 Directional Light 和 Volume，确保为 Visual Elements 提供合适的照明并模拟天气条件. 这些组件的详细描述可以在 这里 找到.

    除了建筑物或绿化等 Visual Elements 外，它还包含负责交通的整个建筑. 流量涉及由 TrafficSimulator 在模拟中生成的 NPCVehicles - 使用流量组件. 下面提供了流量组件的快速概述，但是，您可以阅读它们的详细描述 此处.

    NPCPedestrians 也是 Environment 组件，但它们不受 TrafficSimulator 控制. 他们添加了控制他们移动的脚本 - 你可以阅读更多细节 查看这里.

    交通组件#

    TrafficLanes 和 StopLines 是从 Lanelet2 加载到 Environment 中的元素. TrafficLanes 定义了交叉引用，以便沿车道创建路线. 此外，交叉路口处的每个 TrafficLane 都有特定的让行优先条件. TrafficSimulator 使用 TrafficLanes 来生成 NPCVehicles 并确保它们沿着这些车道移动. 如果某些 TrafficLanes 在交叉路口之前结束，则它引用了 StopLine. 在带有 TrafficLights 的交叉路口，每个 StopLine 都引用最近的 TrafficLight. TrafficLights 属于视觉元素组之一，并提供一个界面来控制模拟交通光源(灯泡)的视觉元素. 单个 TrafficIntersection 负责控制一个路口的所有 TrafficLights. 上述组件的详细说明在 本节 .

    EgoVehicle#

    EgoVehicle 是负责模拟在场景中移动的自动驾驶汽车的组件. 它包括:

    • 与其视觉外观相关的 Models 和 Reflection Probe 组件.
    • Colliders 提供碰撞和在道路上移动的能力.
    • Sensors 提供与车辆状态相关的数据，包括其在 Environment 中的位置和速度以及周围环境的状态.
    • Vehicle 组件，该组件模拟动力学，控制 **Wheel 并负责确保其运动.
    • Vehicle Ros Input 和 Vehicle Keyboard Input 引用 Vehicle 对象并在其中设置控制命令的组件.
    • Vehicle Visual Effect 为 Vehicle 提供控制照明的界面.

    详细说明 EgoVehicle 及其上面提到的组件 查看这里. 描述了默认场景中使用的传感器在 EgoVehicle 上的放置 查看这里. 有关每个传感器的详细信息，请参阅以下部分: Pose, GNSS, LiDAR, IMU, Camera, Vehicle Status.

    FixedUpdate 限制#

    在 AWSIM 中，传感器的发布方法由 FixedUpdate 函数触发，输出频率由:

    time += Time.deltaTime;
var interval = 1.0f / OutputHz;
interval -= 0.00001f; // Allow for accuracy errors.
if (time < interval)
    return;
timer = 0;

    由于此代码在 FixedUpdate 方法中运行，因此必须注意 Time.deltaTime 等于 Fixed Timestep，如 Unity Time.deltaTime 文档 中所述. 因此，每次调用 FixedUpdate 时，传感器脚本中的 time 变量将递增 Fixed Timestep 的常量值，与实时的实际传递无关. 此外，如 Unity 文档 中所述，FixedUpdate 方法可能会在调用 Update 方法之前执行多次，从而导致连续 FixedUpdate 调用之间的时间间隔极小. 下图说明了调用 FixedUpdate 事件函数的机制.

    在每个帧 (游戏 tick) 期间，执行以下作:

    • 首先，Delta Time 计算为当前帧与前一帧之间的差值,
    • 接下来，将 Delta Time 添加到 Time (常规时间),
    • 之后，进行检查以确定 Fixed Time (物理时间)落后于 Time 多少,
    • 如果 Time 和 Fixed Time 之间的差等于或大于 Fixed Timestep，则调用 Fixed Update 事件函数,
    • 如果调用了 FixedUpdate 函数，则 Fixed Timestep 将添加到 Fixed Time 中,
    • 再次执行检查以评估 Fixed Time 落后于 Time 的数量,
    • 如有必要，将再次调用 FixedUpdate 函数,
    • 如果 Time 和 Fixed Time 之间的差异小于 Fixed Timestep，则调用 Update 方法，然后是场景渲染和其他 Unity 事件函数.

    因此，当 FPS(每秒帧数)不稳定或在 FPS、固定时间步长和传感器 OutputHz 的某些组合下，此引擎功能可能会导致意外行为

    在帧速率较低的情况下，建议减小模拟的 Time Scale. Time Scale (时间比例) 值会影响仿真时间，仿真时间是指在模型中模拟的时间，并且可能会也可能不会以与实时相同的速率进行. 因此，通过减小时间尺度，模拟时间的进程会减慢，从而允许模拟有更多时间执行其任务.
